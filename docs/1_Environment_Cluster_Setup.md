# Kubernetes Cluster Environment Setup

## Overview

This document outlines the process of building the Kubernetes environment used throughout this project.

Starting with three Ubuntu virtual machines, I installed Kubernetes using **kubeadm**, configured **containerd** as the container runtime, deployed **Calico** for pod networking, and created a fully functional three-node Kubernetes cluster.

This environment serves as the foundation for the remaining sections of the project, including Kubernetes Dashboard, Prometheus and Grafana monitoring, Harbor Private Registry, and application deployments.

# 1. Preparing the Servers

Before installing Kubernetes, I updated each virtual machine to ensure the operating system was current and that all required packages were available.

```bash
sudo apt update
sudo apt upgrade -y
```

---

# 2. Installing Required Packages

Kubernetes depends on several packages for secure package management and repository configuration.

The following packages were installed on each node.

```bash
sudo apt install -y \
apt-transport-https \
ca-certificates \
curl \
gpg
```

---

# 3. Installing and Configuring containerd

Kubernetes requires a container runtime to create and manage containers.

For this project I used **containerd**, which is the runtime recommended by Kubernetes.

After installation, I generated the default configuration and enabled the **systemd cgroup driver** to ensure compatibility with kubelet.

```bash
sudo apt install -y containerd

sudo mkdir -p /etc/containerd

containerd config default | sudo tee /etc/containerd/config.toml

sudo sed -i \
's/SystemdCgroup = false/SystemdCgroup = true/' \
/etc/containerd/config.toml

sudo systemctl restart containerd

sudo systemctl enable containerd
```

### Verification

Before proceeding, I verified that the runtime had been installed successfully and that the service was running.

```bash
containerd --version

sudo systemctl status containerd
```

---

# 4. Installing Kubernetes Components

The Kubernetes repository was added before installing the core Kubernetes tools:

- kubeadm
- kubelet
- kubectl

These components are responsible for cluster initialization, node management, and cluster administration.

```bash

# (Insert the repository configuration commands used during installation)

sudo apt install -y \
kubeadm \
kubelet \
kubectl

sudo apt-mark hold kubeadm kubelet kubectl
```

### Verification

```bash
kubectl version --client
```

---

# 5. Initializing the Kubernetes Control Plane

The control plane was initialized on the **master** node using kubeadm.

During this process, Kubernetes generated the certificates, created the API server, initialized the control plane components, and produced the join command required for worker nodes.

```bash
sudo kubeadm init \
--pod-network-cidr=192.168.0.0/16
```

---

# 6. Configuring kubectl

Once the control plane was initialized, I configured **kubectl** by copying the administrator configuration file into my home directory.

This allows kubectl to communicate with the Kubernetes API Server without requiring root privileges.

```bash
mkdir -p $HOME/.kube

sudo cp /etc/kubernetes/admin.conf $HOME/.kube/config

sudo chown $(id -u):$(id -g) $HOME/.kube/config
```

### Verification

```bash
kubectl cluster-info
```

---

# 7. Joining the Worker Nodes

Using the join command generated by kubeadm, both worker nodes were added to the cluster.

Once joined, the control plane was able to schedule workloads across all three machines.

```bash
sudo kubeadm join <control-plane-ip>:6443 \
--token <token> \
--discovery-token-ca-cert-hash sha256:<hash>
```

---

# 8. Installing the Calico Network Plugin

Pods running on different nodes require a Container Network Interface (CNI) to communicate.

For this project, I deployed **Calico**, which provides networking between pods as well as support for Kubernetes Network Policies.

```bash
kubectl apply -f https://raw.githubusercontent.com/projectcalico/calico/v3.29.0/manifests/calico.yaml
```

### Verification

```bash
kubectl get pods -n calico-system
```

---

# 9. Verifying the Cluster

After the installation was complete, I verified that:

- All three nodes had joined successfully.
- Kubernetes system pods were healthy.
- The cluster was ready for application deployments.

```bash
kubectl get nodes -o wide

kubectl get pods -A
```
